# 产品需求文档 (Product Specification)

## 项目概述

**项目名称**：A股 AI 实时诊断工具（**AguMaster**）

**一句话描述**：纯前端 Vue 看板（H5 + **Win11 桌面悬浮窗**），按用户所选 **交易所对齐刷新档位**（5 / 15 / 30 / 60 分钟）对 **全池最多 20 只自选**（A 股股票 + 场内 ETF/LOF 混排）拉取实时快照，结合可选持仓成本调用用户自带 Key 的大模型做短线诊断与操作建议（**v2.5** 起拆分为**短期持仓建议 + 波段建议**）；桌面模式以半透明小窗钉选展示 **1–5 只**，**调度仍全池刷新**；**Electron 桌面版**支持 **AI 自动关键位计算 + 价格突破预警 + 高风险 signal 弹窗通知**；**v2.4** 新增 **左侧可折叠分组侧边栏** + **三态刷新开关（关闭 / 正常 / 超频 30s）**。

**目标**：
- 在交易时段内自动、准时完成 **最多 20 只自选**的行情刷新与 AI 诊断
- **AI 自动给出支撑/压力位**，并在价格突破关键位时以桌面弹窗通知
- 零后端部署，配置与数据全部留在本地（浏览器 LocalStorage / 桌面版同源存储）
- 手机 H5 与桌面完整看板均可使用；**Win11 可选半透明悬浮窗**常驻桌面盯盘
- 界面文案 **跟随系统语言**（`zh*` → 简体中文，否则 English）

## 目标用户

**核心用户群体**：内部自用的 A 股/ETF 短线交易者（有基础技术面认知的散户）

**用户特征**：
- 年龄段：25–45 岁
- 技术水平：能自行申请并配置大模型 API Key；能输入股票或场内基金代码
- 使用场景：交易时段盘中盯盘（手机竖屏 / 桌面多列 / **Win11 半透明悬浮小窗 1–5 卡**）；非交易时段偶尔手动刷新复盘
- 痛点：手动逐只复盘耗时长；股票与 ETF 分散在不同软件；希望固定节奏获得结构化 AI 诊断

## 核心功能

### 功能 1：资产配置与 Key 管理

**功能描述**：用户配置大模型连接信息，维护 **统一自选池（最多 20 只，股票与场内基金混排）**，以及 **桌面通知预警设置**，数据持久化在本地。

**输入**：
- `baseUrl`：string，OpenAI 兼容 API 地址，示例 `https://api.deepseek.com/v1`
- `apiKey`：string，用户自有 Key
- `model`：string，模型名称，示例 `deepseek-chat`
- `investmentStyle`（v2.7）：enum，**全局投资风格**，`'aggressive'`（激进）| `'neutral'`（中立，默认）| `'conservative'`（保守）。影响 AI 在风险偏好、建议措辞、调仓频率上的输出倾向，详见功能 4
- `refreshFrequency`：enum，自动刷新档位，**仅允许预设值** `5` | `15` | `30` | `60`（单位：分钟），默认 `30`；**不支持**任意数值或非预设分钟数
- `widgetPinnedCodes`（可选，**仅桌面悬浮模式**）：string[]，从自选池中钉选要在悬浮窗展示的代码，**最少 1 只、最多 5 只**；未配置时悬浮窗不可用
- `widgetOpacity`（可选，桌面）：number，`0.70`–`1.00`，默认 `0.90`；窗口背景不透明度
- `widgetAlwaysOnTop`（可选，桌面）：boolean，默认 `true`；是否始终置顶
- `alertSettings`（可选，**仅供桌面版 Electron**）：`AlertSettings` 对象，控制各类桌面通知的启用/禁用，详见功能 8
- `symbolCode`：string，证券代码，支持 `600519` / `sh600519` / `510500` / `sh510500` / `159915` / `sz159915` 等格式
- `positionQty`（可选）：number，持仓数量，**正整数**，单位：**股**（A 股）/ **份**（场内基金）
- `costPrice`（可选）：number，持仓成本价，**正数**，单位：**元**，最多 **3 位小数**

**输出**：
- LocalStorage / `agu.config.local.json` 持久化的配置对象 `{ baseUrl, apiKey, model, investmentStyle?, refreshFrequency, ... }`
- `WatchlistItem`：
  ```json
  {
    "code": "sh510500",
    "name": "中证500ETF",
    "market": "sh",
    "instrumentType": "fund_etf",
    "positionQty": 1000,
    "costPrice": 6.350,
    "keyLevels": [
      { "price": 6.12, "label": "支撑位", "type": "support", "source": "ai", "updatedAt": "2026-06-12T10:30:00+08:00" },
      { "price": 6.85, "label": "压力位", "type": "resistance", "source": "ai", "updatedAt": "2026-06-12T10:30:00+08:00" },
      { "price": 7.10, "label": "前高", "type": "custom", "source": "manual" }
    ],
    "keyLevelsLocked": false,
    "groupId": "group_1",
    "refreshMode": "normal"
  }
  ```
  - `instrumentType`：`stock`（A 股股票）| `fund_etf`（场内 ETF/LOF）
  - `positionQty`、`costPrice`：**可选**；未填时仅做技术面诊断，不输出持仓操作建议
  - `keyLevels`：**可选，仅供桌面版**；最多 **4 个**价格关键位，AI 自动填充支撑位（`type: "support"`）和压力位（`type: "resistance"`）各 1 个，用户可手动添加最多 2 个自定义警戒价（`type: "custom"`），详见功能 8
  - `keyLevelsLocked`：**可选**，默认 `false`；`true` 时 AI 不再自动更新关键位
  - `groupId`：**可选**（v2.4），所属分组 ID；未设置时归入默认「未分组」
  - `refreshMode`：**可选**（v2.4），枚举 `'off'` | `'normal'` | `'overclock'`，默认 `'normal'`
    - `'off'`：不参与自动定时刷新与超频监控（「立即刷新」和单卡 ↻ 仍可用）
    - `'normal'`：参与主调度自动刷新
    - `'overclock'`：参与主调度 + 独立 30s 高频行情监控（**要求该证券已配置至少 1 个关键位**；同时最多 **5 只**处于此状态，超限时 Toast 提示）

- `UserGroup`（v2.4）：
  ```json
  { "id": "group_1", "name": "短线", "order": 0, "collapsed": false }
  ```
  - `id`：唯一标识
  - `name`：分组名称，≤ 10 字
  - `order`：排序序号
  - `collapsed`：侧边栏中是否折叠

**业务规则**：

#### 自选池范围（v2.3）

| 类型 | 示例代码 | 上限 | 说明 |
|------|----------|------|------|
| A 股股票 + 场内 ETF/LOF | `sh600519`、`sz159915` | **20 只** | 统一计数，股票与基金不分池 |
| 场外开放式基金 | `012345`、`110011` 等 | ❌ | 不支持 |

#### 代码识别规则

- 上海：`6xxxxx` → 股票；`51/56/58xxxx` → 场内基金
- 深圳：`0/3xxxxx` → 股票；`15/16/18xxxx` → 场内基金
- 北京：`4/8xxxxx` → 股票（北交所）
- 自动补全市场前缀（如 `510500` → `sh510500`）
- `instrumentType` 字段保留用于 AI Prompt 区分资产类型，但**不再用于分池计数**

#### 通用规则

- **统一上限**：自选池最多 **20 只**；满池时 Toast「自选已达上限 20 只」
- 重复代码去重，Toast「该证券已在自选池」
- 非法代码或场外基金代码拒绝添加并 Toast 提示具体原因
- **刷新档位**：配置区提供 **下拉单选**（非自由输入），选项固定为下表；切换后立即按新档位重算下一对齐点
- **持仓/成本（可选）**：添加自选时可折叠填写「持仓数量」「成本价」；已添加卡片支持编辑或清空；二者需**同时填写**或**同时为空**；只填一项时 Toast「请同时填写持仓数量与成本价，或留空」
- `positionQty` 上限 **999,999,999**；`costPrice` 范围 **0.001–999,999.999**
- 卡片 Header **现价**按行情原始精度展示，**保留 3 位小数**；**成本价**展示与录入均为 **3 位小数**
- 程序预计算并随 Prompt 下发：`costPrice`、`positionQty`、现价 `price`、浮动盈亏率 `pnlPct`（`(price - costPrice) / costPrice * 100`，保留 2 位小数；`price` 缺失时不计算）
- **桌面悬浮钉选**：`widgetPinnedCodes` 中每只必须存在于当前 `watchlist`；删除自选时自动从钉选中移除；钉选不足 1 只时不允许开启悬浮窗
- API Key、baseUrl、model、refreshFrequency、钉选列表、自选列表以 **明文** 存入 LocalStorage 或本地配置文件
- 支持删除自选（卡片删除按钮）
- 未配置 apiKey 时，允许添加自选与查看上次诊断，自动/手动 AI 跳过并 Banner 提示

#### 自定义分组与侧边栏（v2.4）

- **分组数据结构**：`UserGroup { id, name, order, collapsed }` 存入 `config.groups`
- **默认分组**：系统内建「未分组」（id=`__default__`，不可删除、不可重命名）；所有 `groupId` 为空或未匹配的证券归入此组
- **新建分组**：侧边栏顶部「+ 新建分组」按钮，弹出内联输入框（≤ 10 字）；分组名称不可重复
- **重命名/删除分组**：分组标题右侧 `…` 菜单 → 重命名 / 删除；删除分组时组内证券移入「未分组」，**已删除分组中的证券不丢失**
- **排序**：侧边栏分组可拖拽排序（或 ↑↓ 箭头），影响主页分组显示顺序
- **折叠**：分组默认展开，点击标题可折叠/展开；折叠状态持久化到 `collapsed` 字段
- **移动证券**：拖拽卡片到侧边栏目标分组名称上，修改 `groupId`；或通过卡片右键菜单「移动到分组」

#### 三态刷新开关（v2.4）

- `WatchlistItem.refreshMode`：枚举 `'off'` | `'normal'`（默认）| `'overclock'`
- **卡片图标**：每张卡片 Header 显示三态切换按钮（◌ 关闭 / ⚡ 正常 / ⚡⚡ 超频），点击循环切换
- **超频限制**：
  - 切换至超频时校验：必须有 `keyLevels.length > 0`，否则 Toast「请先配置关键位」
  - 同时最多 5 只处于超频状态，超限时 Toast「超频刷新最多 5 只」
  - 超频 = 主调度正常刷新 + 独立 30s 高频行情拉取 + 价格穿透预警检查
- **关闭状态**：不参与主调度自动刷新，也不参与超频监控；「立即刷新」和单卡 ↻ 仍生效
- **配置面板**：「展开配置」中列出所有自选及其当前刷新模式，可快速切换

**异常处理**：
- LocalStorage 不可用（隐私模式）：顶部 Banner 提示「本地存储不可用，刷新后数据将丢失」
- LocalStorage 配额满：Toast 提示并阻止写入
- 旧版 `StockItem` 无 `instrumentType` 字段：读取时默认补全为 `stock`，向后兼容

**优先级**：高

**状态**：待开发

### 功能 2：行情拉取（实时快照）

**功能描述**：在纯前端环境下，通过 **JSONP** 拉取公开行情接口的实时快照，作为 AI 诊断输入。**行情由程序拉取，LLM 仅做诊断，不参与数据获取。**

**输入**：
- 自选池证券代码列表（最多 **20 只**，股票与基金混排）
- 当前北京时间（用于数据时间戳标注）

**输出**：
- 每只证券的标准化快照对象：
  ```json
  {
    "code": "sh510500",
    "name": "中证500ETF",
    "instrumentType": "fund_etf",
    "price": 6.52,
    "changePct": -0.45,
    "high": 6.58,
    "low": 6.48,
    "volume": 1234567,
    "amount": 9876543210,
    "prevClose": 6.55,
    "quoteTime": "2026-06-09 10:30:05"
  }
  ```

**业务规则**：

#### 场内基金行情说明

- 场内 ETF/LOF 与 A 股共用腾讯/新浪 JSONP 接口，**调用方式不变**
- 部分 ETF 在新浪接口可能返回空数据，依赖腾讯主源；双源均失败则走既有降级逻辑
- 快照新增 `instrumentType` 字段，供 LLM Prompt 区分资产类型

#### 行情源方案（v1 已定稿）

**架构原则**：结构化快照 → LLM 诊断，职责分离；不在 v1 引入后端代理，不采用 LLM 联网抓取行情。

| 层级 | 数据源 | 费用 | 调用方式 | v1 状态 | 说明 |
|------|--------|------|----------|---------|------|
| 主源 | 腾讯财经 `qt.gtimg.cn` | 免费，免 Key | **JSONP** | ✅ 启用 | 纯前端最佳；返回 `v_sh600519="..."`，按 `~` 解析 |
| 备源 1 | 新浪财经 `hq.sinajs.cn` | 免费，免 Key | **JSONP** | ✅ 启用 | 主源失败时切换；部分环境 Referer 限制 |
| 备源 2 | 东方财富 `push2.eastmoney.com` | 免费，免 Key | `fetch` + 代理 | ⏸ v1 不启用 | 接口本身免费，但浏览器 CORS 受限；**仅在未来引入 Cloudflare Worker 等代理时可启用** |
| 排除 | 同花顺 QuantAPI | 付费 | 官方 REST | ❌ 不采用 | 需签约付费，违背零成本自用定位 |
| 排除 | LLM 联网抓取（同花顺/东财网页等） | — | — | ❌ 不采用 | 不可控、易幻觉、延迟与 Token 成本高 |

> **「免费」说明**：腾讯/新浪/东财公开接口均为网页同源数据，**无需注册或 API Key**，但不构成商业授权；本产品定位为**内部自用**，若未来对外商用须更换为有合同的数据服务（Wind、同花顺官方 API 等）。

#### 拉取与降级流程

1. 主源腾讯 JSONP 批量请求（**每批最多 5 只**；20 只自选时分 4 批串行，批间隔 ≥ 300ms）
2. 主源失败 → 等待 ≥ 500ms → 备源 1 新浪 JSONP 逐只或批量重试
3. 备源 1 仍失败 → 该卡遮罩「行情获取失败」，**不触发** AI（节省 Token）
4. 备源 2 东财：**v1 不实现**；v1.x 若用户自建代理，可在 `quoteJsonp.ts` 扩展第三层降级

#### 数据范围

- **v1 仅使用实时快照**，不拉取历史 30 分钟 K 线序列；AI 基于「当前时刻快照 + 对齐时刻标签」做短线诊断
- 9:15–9:30 集合竞价：成交量字段替换为 `"集合竞价阶段"`，Prompt 告知 AI 忽略量价配合分析

**异常处理**：
- 主源失败 → 自动切换备源 1 重试 1 次
- 主备均失败：该卡片遮罩「行情获取失败」，**不触发** AI 请求
- 部分失败：失败卡单独遮罩，成功卡正常进入 AI 流程
- 解析字段缺失（如停牌）：该字段填 `null`，Prompt 标注「停牌/无成交」

**优先级**：高

**状态**：待开发

### 功能 3：定时调度与手动刷新

**功能描述**：交易时段内按用户所选 **刷新档位**，在对应的 **交易所对齐时刻** 自动触发全量刷新；非交易时段停止自动调度，仅允许手动刷新。

**输入**：
- 系统当前时间（北京时间）
- 用户配置的 `refreshFrequency`（`5` | `15` | `30` | `60`，默认 `30`）
- 用户点击「立即刷新」

**输出**：
- 触发一轮「行情拉取 → AI 诊断」流水线
- 更新各卡片 Footer 的「更新于 HH:mm:ss」时间戳

**业务规则**：

#### 刷新档位（v2.1，预设枚举，不对齐网格的数值一律不提供）

| 档位值 | UI 文案 | 对齐自动触发点（交易日） | 每日自动次数 | 定位 |
|--------|---------|--------------------------|--------------|------|
| `5` | **5 分钟 · 高频** | 09:30 起每 5 分钟至 11:30 / 13:00 起至 15:00 | **50 次** | 最密档位；Token 消耗最高，适合极短线盯盘 |
| `15` | **15 分钟 · 短线** | 09:30 起每 15 分钟至 11:30 / 13:00 起至 15:00 | **18 次** | 介于 5 与 30 之间，适合 15m 级别策略 |
| `30` | **30 分钟 · 标准**（默认） | 09:30、10:00、10:30、11:00、11:30、13:00、13:30、14:00、14:30、15:00 | **10 次** | 默认档位；与 v1 原始网格一致 |
| `60` | **60 分钟 · 省 Token** | 09:30、10:30、11:30、13:30、14:30 | **5 次** | **最低频 / 最小性价比**；适合持仓稳定、以省 API 费用为主 |

**桌位设计约束（已定稿）**：
- **不提供** 45 / 任意分钟数：无法从 09:30 起稳定对齐交易所网格，调度实现与用户预期均不可控
- **5 / 15 分钟档**：从 09:30 起按固定步长生成上午 / 下午对齐点；用户需自行权衡 Token 成本（20 只串行 LLM）
- **最小性价比档 = 60 分钟**：在仍覆盖每个交易小时的前提下，将每日自动 LLM 调用量降至 30 分钟档的 **50%**

#### 交易时段
- 上午：09:30–11:30（**交易日**内）
- 下午：13:00–15:00（**交易日**内）
- 午休 11:30–13:00：**停止**自动调度
- 非交易日（含法定节假日、周末非调休日）：**停止**自动调度

#### 交易日判定（依赖功能 6 日历）
1. 若本地已缓存当年日历，按国务院节假日安排判定：
   - `isOffDay: true` → **非交易日**（法定放假）
   - `isOffDay: false` → **交易日**（调休补班，含原本为周末的日期）
2. 若日历未加载或拉取失败，**降级**为「周一至周五 = 交易日」
3. 在交易日基础上，再叠加交易时段（09:30–11:30、13:00–15:00）决定是否自动调度

#### 对齐时刻（基准网格，30 分钟粒度）

基准网格（`refreshFrequency = 30` 时全部启用）：
09:30、10:00、10:30、11:00、11:30、13:00、13:30、14:00、14:30、15:00

`refreshFrequency = 60` 时，从基准网格中**仅保留整点后半段对齐点**：
09:30、10:30、11:30、13:30、14:30（**跳过** 10:00、11:00、13:00、14:00、15:00）

`refreshFrequency = 15` / `5` 时，分别在上午 09:30–11:30、下午 13:00–15:00 内按 **15 / 5 分钟步长**生成对齐点（含 11:30、15:00 收市点）。

#### 调度逻辑
1. 页面加载时：先确保当年交易日历已就绪（见功能 6）；若当前处于交易时段，**立即执行一轮**全量刷新，然后注册**当前档位**下的下一个对齐时刻定时器
2. 对齐算法：在**当前档位允许的对齐点集合**中，计算当前时间之后的下一个点；若已过当日最后一个对齐点则等到**下一交易日**该档首个对齐点（均为 09:30）
3. 切换 `refreshFrequency`：立即按新档位重算下一对齐点，**不** retroactive 补跑已错过的更密档位时刻
4. **手动刷新**：任意时间可点击；立即执行全量刷新（**所有 20 只均刷新，无视 refreshMode**），并重置自动调度计时器
5. **自动刷新范围**：对齐时刻自动触发时刷新全池 20 只（`refreshMode !== 'off'` 的证券）；单卡 ↻ 无视 refreshMode 始终刷新
6. **超频调度（v2.4）**：若存在 `refreshMode === 'overclock'` 的证券，开启独立 30s 高频轮询（仅拉行情 + 更新卡片价格 + 关键位穿透预警，**不触发 AI 诊断**）
6. 非交易时段手动刷新：照常拉行情与调 AI（盘前/盘后/节假日数据可能为昨收或静态，Prompt 标注 `session: "非交易时段"`）
7. 浏览器 H5 后台 Tab：使用 `setTimeout` 链式调度；回到前台时**仅重算**下一对齐点，**不**因错过对齐点自动补跑（见 ADR-006 浏览器分支）
8. **桌面 Electron 模式**（功能 7）：窗口最小化至托盘或悬浮窗失焦时，调度**不中断**；对齐时刻触发时仍执行 **全池（最多 20 只，按 autoRefresh 筛选）** 行情 + AI 流水线，与完整看板一致

**异常处理**：
- 上一轮尚未完成时用户再次点击刷新：忽略重复点击，按钮置灰并显示「刷新中…」

**优先级**：高

**状态**：待开发

### 功能 4：大模型诊断

**功能描述**：将实时快照（及可选持仓成本）拼装为 Prompt，调用 OpenAI 兼容 API，返回结构化 JSON 诊断结果。**v2.7 起注入全局投资风格参数，AI 按风格调整风险偏好与建议措辞；User Message 增加日内振幅、相对高低点位置、量价方向等上下文，要求 AI 给出前瞻性趋势判断。**

输出路径（v2.6 两路，v2.7 不变）：

- **有持仓**：输出 `signal` + `analysis` + `risk` + `shortAction` + `bandAction` + `supportLevel` + `resistanceLevel`
- **无持仓（观望状态）**：输出 `signal` + `analysis` + `risk` + `buildPositionAdvice` + `supportLevel` + `resistanceLevel`；**不输出** `shortAction` / `bandAction`。无持仓视为用户尚未买入该只证券，需要的是"该不该建仓/在什么价位建仓"，而非"如何操作已有仓位"。

**输入**：
- 单只证券快照对象（含 `instrumentType`、**v2.7 新增**：日内振幅 `amplitude`、当前价相对于高低点的位置比例 `positionInRange`、较昨收的涨跌方向 `direction`）
- 对齐时刻标签，示例 `2026-06-09 10:30`
- 时段标签：`交易中` / `集合竞价` / `非交易时段`
- 用户配置的 baseUrl、apiKey、model
- **v2.7 新增**：`investmentStyle`（`'aggressive'` | `'neutral'` | `'conservative'`），默认 `'neutral'`
- 可选持仓上下文：`positionQty`、`costPrice`、预计算 `pnlPct`（见功能 1）

**输出（有持仓）**：
```json
{
  "signal": "逢高减仓",
  "analysis": "放量滞涨，上影压制明显",
  "risk": "30分钟内回落风险偏高",
  "shortAction": "现价高于成本约2%，可减半仓锁定利润",
  "bandAction": "接近阻力位6.85，未突破前不宜追高，回落至6.12支撑附近可分批关注",
  "supportLevel": 6.12,
  "resistanceLevel": 6.85
}
```

**输出（无持仓，观望状态）**：
```json
{
  "signal": "空仓观望",
  "analysis": "缩量回踩支撑位，均线附近整理",
  "risk": "若放量跌破6.12支撑，短线走弱确认",
  "buildPositionAdvice": "现价贴近支撑位6.12附近，可等待放量企稳后轻仓试多，止损设于6.00下方",
  "supportLevel": 6.12,
  "resistanceLevel": 6.85
}
```

**字段定义**：
- `signal`：4 字操盘提示，必须为**固定短语**（见 UI-Spec 附录 A signal 词库）
- `analysis`：≤ 50 字短线技术形态诊断
- `risk`：≤ 30 字风险提示
- `shortAction`：**仅有持仓时必填**；**≤ 30 字**；须结合 `pnlPct` 与 `signal` 给出**可执行**的短期持仓动作；禁止空泛表述
- `bandAction`：**仅当该证券配置了持仓时必填**（v2.6）；**≤ 30 字**；**必须锚定 `supportLevel` / `resistanceLevel`** 与当前价相对位置给出数日级别波段建议；禁止脱离关键位凭空给趋势观点
- `buildPositionAdvice`：**仅当该证券未配置持仓时必填**（v2.6，替代 v2.5 的 bandAction）；**≤ 30 字**；**必须锚定 `supportLevel` / `resistanceLevel`**，给出建仓方向与建议价位（如「可等待放量企稳支撑位后轻仓试多」「接近阻力位暂不建议追高建仓」）；禁止脱离关键位凭空给出买卖建议
- `supportLevel`：**必填**，技术支撑位
- `resistanceLevel`：**必填**，技术压力位
- **互斥规则**：`shortAction` / `bandAction` 与 `buildPositionAdvice` **互斥**，一个 Prompt 输出中不可能同时出现。有持仓 → 输出前两者；无持仓 → 输出后者。

**业务规则**：
- 最多 20 只证券 **串行**调用 LLM（间隔 ≥ 300ms），避免 429
- 请求体启用 `response_format: { "type": "json_object" }`；`temperature: 0`（降低同输入下的随机漂移）
- **快照指纹去重（v1.6）**：对 `code + session + price + changePct + high + low + volume + prevClose` 计算指纹；若与上次成功诊断指纹**一致**，**跳过 LLM**，沿用缓存结论
- 休市后多次手动刷新：行情快照不变 → 结论**必须不变**；卡片 Footer 显示「行情未变·沿用 HH:mm:ss」
- 指纹变化（盘中现价变动）→ 正常重新调用 LLM
- System Prompt 模板见 [UI-Spec.md](./UI-Spec.md) 附录 A；User Message 含 `instrumentType` + **v2.7 增强上下文**（振幅、相对高低位置、较昨收涨跌方向）；有持仓时追加 `positionQty`、`costPrice`、`pnlPct`；**v2.7 起 User Message 额外注入 `investmentStyle` 参数，System Prompt 含三档风格指令**（见下方 Style Directives）
- **v2.7 投资风格指令（Style Directives）**，按所选风格注入 System Prompt：
  - **激进（aggressive）**：偏好短线积极操作，放宽减仓阈值（pnlPct > 8% 方考虑），优先建议逢低加仓/做T，波段建议可给出更积极的目标位
  - **中立（neutral）**：平衡风险收益，按 v2.6 的 pnlPct > 5% 减仓阈值，波段建议锚定关键位 ± 1 个标准差
  - **保守（conservative）**：以保本和风控为主，降低风险敞口，减仓阈值收紧至 pnlPct > 3%，波段建议偏谨慎（以防御性建议为主）
- **v2.7 前瞻性要求**：Prompt 要求 AI 结合日内振幅与量价方向，推断当日最可能的走势方向（偏多/偏空/震荡），并在 bandAction 中给出未来数日的波段趋势预判（不强制 7 日，以支撑阻力位可推导的合理区间为准）
- `signal` 配色规则见 UI-Spec；卡片操作建议展示规则（v2.6）：
  - **有持仓**：Content 区 Risk 行下方依次展示「波段建议：{bandAction}」「短期建议：{shortAction}」
  - **无持仓（观望）**：Content 区 Risk 行下方展示「建仓建议：{buildPositionAdvice}」，不渲染 bandAction / shortAction 行
  - 两套展示**互斥**，取决于该证券是否配置持仓
- **关键位自动填充（v2.2）**：LLM 返回的 `supportLevel` / `resistanceLevel` 在解析成功后自动写入 `WatchlistItem.keyLevels`（分别对应 `type: "support"` 和 `type: "resistance"`），受偏差保护规则约束（见功能 8）
- 诊断结果与指纹一并写入 LocalStorage（`agu_diagnosis_cache`）

**异常处理**：
- 指纹命中缓存：跳过 LLM，不消耗 Token；全池命中时 Toast「行情未变，已沿用缓存诊断」
- `JSON.parse` 失败：Signal 条显示「解析失败」，Content 区展示 Raw 文本；**自动对该证券重试刷新**（间隔递增，最多 30 次），直至解析成功或遇到行情失败 / 鉴权失败
- 有持仓但 LLM 未返回 `shortAction` 或 `bandAction`：视为解析失败，走 Raw 展示分支
- **LLM 未返回 `buildPositionAdvice`（无持仓时），或 `buildPositionAdvice` / `bandAction` 未引用 `supportLevel` / `resistanceLevel`**：视为解析失败，走 Raw 展示分支（v2.6；锚定校验由 Prompt 强约束 + 解析层校验双重保障）
- HTTP 401 / 429：顶部 Toast「API Key 校验失败或额度不足，请检查配置」，停止本轮剩余证券的 AI 调用
- 单只失败不影响其他只；失败卡 Signal 条显示「诊断失败」

**优先级**：高

**状态**：待开发

### 功能 6：交易日历同步

**功能描述**：每年自动从互联网拉取国务院发布的法定节假日与调休安排，缓存到本地，供定时调度准确识别 A 股交易日。

**输入**：
- 当前北京时间（判定年份与是否触发 1 月 1 日刷新）
- 远程节假日 JSON（见下方数据源）

**输出**：
- LocalStorage 缓存对象 `agu_trading_calendar_{year}`：
  ```json
  {
    "year": 2026,
    "fetchedAt": "2026-01-01T09:00:00+08:00",
    "source": "holiday-cn",
    "days": [
      { "date": "2026-01-01", "name": "元旦", "isOffDay": true },
      { "date": "2026-01-04", "name": "春节", "isOffDay": false }
    ]
  }
  ```
- 内存方法 `isTradingDay(date: Date): boolean`

**业务规则**：

#### 数据源（浏览器 `fetch` 可用）

| 层级 | 数据源 | URL 模板 | 说明 |
|------|--------|----------|------|
| 主源 | holiday-cn（国务院公告） | `https://cdn.jsdelivr.net/gh/NateScarlet/holiday-cn@master/{year}.json` | JSDelivr CDN，支持 CORS；数据含放假与调休 |
| 备源 | timor.tech | `https://timor.tech/api/holiday/year/{year}/` | 主源失败时重试 1 次 |

#### 自动拉取触发条件（满足任一即执行）
1. **每年 1 月 1 日**：App 启动或跨日检测（00:00 北京时间）时，若当日为 1 月 1 日且本地 `{year}` 日历的 `fetchedAt` 不是今年，**强制拉取当年日历**
2. **首次缺失**：本地无当前年份缓存时，App 启动立即拉取
3. **跨年**：检测到 `stored.year !== currentYear` 时，拉取新年份日历（如 12/31 夜挂页至 1/1 的场景）

#### 缓存策略
- 当年日历拉取成功后写入 LocalStorage，**整年有效**，不重复请求
- 保留最近 2 年数据；更早年份自动清理
- 配置区展示轻量状态：`交易日历 2026 已同步` / `同步中…` / `同步失败（已降级为周一至周五）`

#### 与 A 股交易日的映射
- A 股休市安排与国务院节假日一致；**调休补班日（`isOffDay: false`）视为交易日**
- 日历仅决定「是否为交易日」；具体开收盘时刻仍由功能 3 的交易时段规则控制

**异常处理**：
- 主源 + 备源均失败：Toast「交易日历同步失败，已降级为周一至周五规则」；调度模块使用降级逻辑
- JSON 解析失败：同失败处理，不阻塞页面其余功能
- 当年国务院尚未发布完整安排（如 1 月初数据不全）：使用已获取部分 + 降级规则；国务院更新后用户可在配置区点击「重新同步日历」（手动兜底）

**优先级**：高

**状态**：待开发

### 功能 7：Win11 桌面悬浮模式（Electron）

**功能描述**：在 Windows 11 上以 **Electron** 打包运行同一 Vue 应用，提供 **无边框半透明、始终置顶（可关）** 的悬浮窗；窗内仅展示用户钉选的 **1–5 只** **标准诊断卡片**（与完整看板同组件、同字段、同 signal 配色），**自动/手动刷新逻辑与 H5 一致且始终针对全池 20 只**，悬浮窗为只读展示（隐藏删除与持仓编辑）。

**输入**：
- 用户配置的 `widgetPinnedCodes`（1–5 只，须为自选池子集）
- `widgetOpacity`、`widgetAlwaysOnTop`
- 与 H5 相同的 scheduler 触发（对齐时刻 / 手动刷新）
- 托盘菜单操作：显示/隐藏悬浮窗、打开完整看板、退出

**输出**：
- 桌面悬浮窗 UI（**复用完整看板 `StockCard`，`widgetMode` 只读**）
- 系统托盘图标与菜单
- 可选：独立「完整看板」大窗（同进程第二 BrowserWindow，或默认尺寸主窗）

**业务规则**：

#### 平台与打包
- **v1.8 范围**：**Windows 11 x64** 先行；macOS / Win10 不在 v1.8 承诺范围
- 安装包：`agu-desktop`（Electron）；H5 浏览器版**继续保留**，互不替代
- 业务代码复用 `web/`；Electron 主进程负责窗口生命周期与托盘

#### 悬浮窗窗口
- 默认尺寸：**420 × 560 px**（可拖拽边缘缩放，最小 360 × 480）
- 样式：**无边框**、`transparent: true`、暗色毛玻璃底板（保证 signal 红绿灰可读）
- 默认透明度：**90%**（`widgetOpacity = 0.90`）；用户可在设置滑块调节 **70%–100%**
- **始终置顶**默认开启，可在设置关闭
- 支持拖拽移动；**可选**吸附屏幕四边（v1.8 至少支持自由拖拽）
- 关闭按钮行为：**最小化到托盘**，不退出进程（托盘菜单提供「退出」）

#### 悬浮窗卡片（与完整看板一致）
- 仅展示 `widgetPinnedCodes` 对应证券，顺序与钉选配置一致
- **复用**完整看板 `StockCard`（`widgetMode=true`）：Header、持仓摘要（若有）、signal 四档条、analysis / risk / 波段建议（bandAction，必有）/ 短期建议（shortAction，仅持仓时）、Footer；**signal / 涨跌幅配色与完整看板相同**
- 悬浮窗内 **隐藏**删除自选与持仓编辑；保留单卡 ↻（**仅刷新该证券**）
- 加载态 / 行情失败 / 解析失败：与完整看板相同遮罩与文案
- 窗内不提供添加自选；配置入口：「打开完整看板」或托盘菜单

#### 刷新与数据
- 对齐时刻自动刷新、配置区「立即刷新」：**刷新全池最多 20 只**（行情 + AI），与功能 3 一致
- 单卡 ↻（完整看板 / 悬浮窗）：**仅刷新该证券**；桌面悬浮窗通过 IPC 委托完整看板主进程执行，完成后 live-sync 回悬浮窗
- 悬浮窗**仅订阅展示**钉选 1–5 只的 diagnosis 结果；全池自动刷新仍更新数据存储，切到完整看板时数据已就绪
- 最小化至托盘 / 悬浮窗被遮挡：**不降低**全池自动刷新范围

#### 钉选配置
- 配置区（完整看板）新增「悬浮窗显示」多选，从当前自选池勾选 **1–5 只**

### 功能 8：桌面通知预警系统（Electron 专属）

**功能描述**：在 Electron 桌面版中，AI 诊断完成后自动计算关键位并检测是否需要弹窗通知；当 signal 命中高风险词库、价格突破关键位、或出现 API/行情异常时，以独立小卡片窗口在屏幕右下角弹出提醒。H5 浏览器版**不支持**此功能。

**输入**：
- 本次全量刷新 / 单卡刷新的所有诊断结果（含 `signal`、`supportLevel`、`resistanceLevel`）
- 各证券当前 `keyLevels` 列表（含偏差保护阈值）
- 用户 `alertSettings`（各类通知开关）
- 当前行情快照 `price`（用于关键位突破判定）

**输出**：
- Electron 主进程创建的独立 `BrowserWindow` 弹窗（无边框、半透明、右下角定位）
- 弹窗内容以 360×200 小卡片展示触发信息，15 秒自动消失或点击关闭
- 多条通知时合并为一个弹窗，按优先级排序

**业务规则**：

#### 通知类型与触发条件

| 类型 | 配置字段 | 触发条件 | 优先级 |
|------|----------|----------|--------|
| **价格关键位突破** | `alertSettings.priceAlert` | 当前 `price` 从上向下穿过关键位或从下向上穿过关键位（穿透判定，非瞬间触碰） | **最高** |
| **AI Signal 高风险预警** | `alertSettings.signalAlert` | 本次诊断 `signal` 命中固定高风险词库：`止损` / `逢高减仓` / `减仓` / `清仓` | **高** |
| **API Key 鉴权失败** | `alertSettings.authErrorAlert` | LLM 调用返回 HTTP 401 / 429，本轮刷新被中止 | **中** |
| **行情全部失败** | `alertSettings.quoteErrorAlert` | 本轮刷新中所有证券行情均拉取失败（主源 + 备源） | **低** |

- `alertSettings.enabled` 为总开关；`false` 时所有通知静默
- 各类通知的独立开关仅在 `alertSettings.enabled = true` 时生效
- 默认值：总开关 `true`，四类子开关均 `true`

#### AI 自动关键位计算与偏差保护

1. **首次提取**：LLM 首次诊断返回 `supportLevel` / `resistanceLevel` 后，自动写入 `WatchlistItem.keyLevels`（`type: "support"` | `type: "resistance"`，`source: "ai"`），记录 `updatedAt`
2. **后续更新**：每次诊断成功后，对比新值与旧值：
   - 偏差 ≤ ±3%：**自动更新**关键位值，更新 `updatedAt`
   - 偏差 > ±3%：**暂停更新**，弹出确认通知「{证券名} 关键位偏差较大（支撑 {旧}→{新} / 压力 {旧}→{新}），是否接受更新？」
3. **用户锁定**：用户在卡片中可点击「锁定关键位」，设置 `keyLevelsLocked = true`；此后 AI 不再自动更新，仅展示当前锁定值
4. **解锁**：用户可手动解锁（`keyLevelsLocked = false`），恢复 AI 自动更新
5. **手动编辑**：用户可手动添加最多 2 个自定义警戒价（`type: "custom"`，`source: "manual"`），与 AI 关键位共存共 4 个槽位
6. **自定义关键位删除**：用户可删除 `source: "manual"` 的关键位；AI 生成的支撑/压力位不可直接删除（但可锁定后手动修改覆盖）

#### 价格突破穿透判定

- **必要条件**：该证券配置了至少 1 个 `keyLevels` 且对应通知开关为 `true`
- **穿透判定**：记录上一轮检查的价格 `lastPrice`，当 `lastPrice` 和当前 `price` 分别位于关键位的两侧时，视为「穿透」
  - 例：`lastPrice = 6.40`，`price = 6.38`，支撑位 = `6.40` → 击穿支撑位，触发预警
  - 例：`lastPrice = 6.35`，`price = 6.35`，支撑位 = `6.40` → 价格仍在支撑位下方但未再次穿透，**不触发**
- **避免重复触发**：同一关键位仅触发一次记录穿透状态，直到价格从另一方向再次穿过该关键位后才重置
- **取消触发**：用户手动修改或删除关键位时，清除对应的穿透状态

#### 超频高频监控（v2.4 重新设计）

- **触发条件**：任意证券 `refreshMode === 'overclock'` 且处于交易时段 → 开启独立 30s 高频轮询
- **轮询范围**：仅拉取所有 `refreshMode === 'overclock'` 的证券（上限 5 只）
- **功能**：仅拉行情 + 更新卡片 snapshot + 关键位穿透预警判定，**不触发 AI 诊断**
- **失败静默**：独立轮询失败跳过当轮，不弹窗
- **与主调度解耦**：独立轮询与主调度完全独立，不受 `refreshFrequency` 档位影响

#### 弹窗窗口规范

- **窗口尺寸**：360 × 200 px
- **位置**：屏幕右下角，距右边缘 20px、底边缘 20px
- **样式**：无边框、半透明（opacity 0.92）、暗色背景、圆角 10px
- **生命周期**：
  - 弹出后 15 秒自动关闭
  - 用户点击任意位置立即关闭
  - 若弹出期间有新通知触发，关闭当前弹窗并重新弹出合并后的新通知列表
- **单条通知内容**：证券名称 · signal（若有）| 关键位类型：价格 → 现价
- **多条通知合并**：标题「{N} 条新预警」，列表按优先级排序
- **按钮**：「查看完整看板」（聚焦主窗口）、「忽略」（关闭弹窗）

#### 配置面板 UI

- 完整看板「展开配置 → 桌面悬浮窗」区域下方新增「桌面通知预警」子区域
- 总开关：勾选框「启用桌面通知」
- 子开关（总开关开启时可见）：
  - ✅ 价格关键位突破
  - ✅ AI 高风险信号预警
  - ✅ API Key 异常
  - ✅ 行情获取异常
- 各开关即时生效（emit `save`），非 Modal

#### 投资风格配置（v2.7）

- 在 API 连接区域（baseUrl / apiKey / model）下方新增「投资风格」下拉选择
- 三档选项：激进 | 中立（默认） | 保守
- 即时生效，无单独保存按钮
- 选择后注入 System Prompt（见功能 4），影响所有证券的 AI 输出倾向

**异常处理**：
- Electron 弹窗创建失败（如 GPU 渲染异常）：降级为托盘区气泡通知，并 Toast 提示
- 偏差保护弹窗出现时 30 秒内用户无操作：自动接受旧值（不更新），关闭弹窗
- 同一证券在 10 秒内连续触发同一关键位穿透：合并为 1 条通知（防抖）

**优先级**：高（桌面版专属）

**状态**：待开发

## AI增强功能

本产品以 AI 诊断为核心能力，以下为 v1 已确认的 AI 功能定义。

### AI增强 1：实时快照短线诊断

**AI能力描述**：根据单只证券（股票或场内基金）实时快照 + 全局投资风格，输出 5 档操盘提示、形态诊断、风险预测、前瞻性波段/短期建议（v2.7 增强）。

**触发条件**：
- 交易时段内，按用户所选刷新档位（5 / 15 / 30 / 60 分钟）的对齐时刻自动触发
- 用户手动刷新
- 页面加载时（交易时段内）

**输入**：System Prompt + User Message（含快照 JSON、对齐时刻；有持仓时含成本与盈亏率）

**输出**：`{ signal, analysis, risk, bandAction [, shortAction] }` JSON

**示例**：
```
User: 【中证500ETF sh510500 | 场内基金】对齐时刻 2026-06-09 10:30 | 交易中
快照: price=6.52, changePct=-0.45%, high=6.58, low=6.48, volume=1234567
持仓: qty=1000, cost=6.35, pnlPct=+2.68%

AI: {"signal":"持股观望","analysis":"缩量震荡，均线附近整理","risk":"跟踪误差放大，谨防贴水扩大","shortAction":"浮盈有限，可继续持有，跌破成本再减仓","bandAction":"现价贴近阻力位，突破前不宜加仓，回踩支撑可分批关注"}
```

### AI增强 2：特殊时段 Prompt 自适应（v1 内置）

**AI能力描述**：集合竞价与非交易时段自动修改 Prompt 约束，避免 AI 对无效数据过度解读。

**触发条件**：09:15–09:30 或非交易时段刷新

**输入**：在 User Message 追加 `session` 字段

**输出**：同 AI增强 1

**示例**：`session: "集合竞价"` → AI 被告知忽略量价配合分析

### AI增强 3：关键位自动计算与预警（v2.2，桌面专属）

**AI能力描述**：每次短线诊断时，LLM 同时输出基于当前快照的技术支撑位与压力位；程序自动将关键位写入证券配置并启动价格监控，突破时弹窗通知。

**触发条件**：
- 每次 AI 诊断成功后自动执行
- 价格穿透判定由独立高频监控（30s）持续检查

**输入**：System Prompt 扩展要求（见 UI-Spec 附录 A v2.2 版）— 要求 LLM 额外输出 `supportLevel` / `resistanceLevel`

**输出**：`{ signal, analysis, risk, supportLevel, resistanceLevel, bandAction [, shortAction] }` JSON

**偏差保护**：新关键位与旧值偏差 > ±3% 时暂停自动更新，弹窗请求用户确认

**示例**：
```
User: 【中证500ETF sh510500 | 场内基金】对齐时刻 2026-06-12 10:30 | 交易中
快照: price=6.52, changePct=-0.45%, high=6.58, low=6.48

AI: {"signal":"持股观望","analysis":"缩量震荡","risk":"跟踪误差放大","supportLevel":6.35,"resistanceLevel":6.70}
→ 程序自动写入 keyLevels：支撑 6.35、压力 6.70
```

## 附录

### 架构决策：行情数据源（ADR-001）

**背景**：曾评估「LLM 自行抓取同花顺」「换东财/同花顺公开接口」等方案。

**决策**：
1. v1 维持 **腾讯 JSONP 主源 + 新浪 JSONP 备源**，不引入后端
2. **禁止** LLM 联网抓取行情；LLM 输入必须是程序解析后的结构化快照
3. 东财接口虽免费免 Key，但纯前端 `fetch` 不可用，**v1 不接入**；若后续加 Cloudflare Worker 代理，作为备源 2 评估
4. 同花顺官方 QuantAPI 为付费服务，**不纳入** v1 范围

**理由**：在「零后端、零数据成本、最多 10 只/30 分钟」约束下，腾讯/新浪 JSONP 是免费且浏览器可用的最优组合。

### 架构决策：自选范围与基金支持（ADR-002）

**背景**：用户要求除股票外，基金也需支持同等 AI 诊断能力。

**冲突**：
- **场外开放式基金**仅每日公布净值，无盘中实时快照，与「30 分钟对齐诊断」架构矛盾
- 股票 5 + 基金 5 共 10 卡，移动端滚动增加，但仍可接受

**决策（v1.5 / v2.3 修订）**：
1. 支持 **场内 ETF/LOF**（与 A 股同交易时段、同 JSONP 行情）
2. **不支持场外开放式基金**
3. **统一计数**：股票与场内基金混排，合计最多 **20 只**（v2.3 由 10 只扩展）
4. 数据结构 `WatchlistItem` 含 `instrumentType`（保留用于 AI Prompt 区分资产类型，但不再用于分池计数）

**理由**：场内基金与现有架构兼容；分池满足用户同时盯 5 股 + 5 基的需求。

### 架构决策：休市诊断一致性（ADR-003）

**背景**：休市后多次刷新，行情快照不变但 LLM 每次返回不同结论，损害可信度。

**根因**：
1. 大模型默认存在随机性（temperature > 0）
2. 休市后仍对**相同快照**重复调用 LLM

**决策**：
1. 引入**快照指纹**；指纹不变则**禁止**重复调用 LLM，沿用上次诊断
2. API 请求固定 `temperature: 0`
3. UI 明示缓存沿用状态，避免用户误以为「又做了一次新分析」

**理由**：诊断应随**行情变**而变，不应随**刷新次数**而变。

### 架构决策：刷新档位预设（ADR-004）

**背景**：用户希望可配置刷新频率；初稿为 5–60 分钟任意整数。

**冲突**：
- 任意分钟数无法从 09:30 起稳定对齐交易所网格（如 45 分钟会在午后漂移）
- 过快档位（5 / 15 分钟）在 10 只串行 LLM 下 Token 成本高，且休市/横盘时大量调用被指纹去重跳过，**性价比极低**

**决策（v1.7）**：
1. **废弃** 5–60 连续可配方案
2. 初版仅提供 **2 档预设枚举**：`60`（最小性价比 / 省 Token）、`30`（默认 / 标准）
3. 调度器从基准 30 分钟网格中按档位**筛选**对齐点，不引入独立 cron 表达式

**修订（v2.1）**：在保留预设枚举约束前提下，新增 **`5` / `15` 分钟**两档；步长档位由 09:30 起在交易时段内程序化生成对齐点。用户选用快档时需自行承担更高 Token 成本。

### 架构决策：持仓成本与操作建议（ADR-005）

**背景**：用户希望在添加自选时可选填持仓与成本，并由 AI 给出操作建议。

**决策**：
1. `positionQty` + `costPrice` **成对可选**；缺一则不做持仓维度分析
2. 程序预计算 `pnlPct` 注入 Prompt，避免 LLM 算术错误
3. 输出新增 `action` 字段（≤30 字），仅在有持仓时必填
4. UI 展示浮动盈亏与操作建议；Footer 仍保留合规声明

**约束**：`action` 为 AI 生成内容，**不构成投资建议**；与现有 Footer 免责声明并存。

### 架构决策：短期建议与波段建议拆分（ADR-008，v2.5）

**背景**：用户反馈单一 `action` 建议过于单薄，希望丰富操作建议。但 AguMaster 是 5 分钟刷新的**短线工具**，无周线/月线/财报数据，"长期建议"缺乏依据。

**冲突**：
- **"长期" vs 产品定位**：产品一句话描述为"短线诊断"，"长期建议"在仅有实时快照的场景下**无数据支撑**，AI 易凭空编造趋势观点误导用户
- **空仓用户功能失效**：原 `action` 仅在有持仓时输出，空仓用户看不到任何操作建议
- **卡片密度**：现有卡片已含 signal/analysis/risk/support/resistance 5 块，再加建议易爆版，悬浮窗 360px 宽放不下

**决策**：
1. **不做"长期建议"**，改为**波段建议**（`bandAction`，数日级别），语义与产品定位自洽
2. 原 `action` 字段**重命名为 `shortAction`**（短期持仓动作），语义不变，仍仅在有持仓时必填
3. 新增 `bandAction`（≤30 字），**无论是否配置持仓均必填**；**必须锚定 `supportLevel` / `resistanceLevel` 与当前价相对位置**，Prompt 强约束 + 解析层校验，禁止脱离关键位的趋势空谈
4. `signal`（4 字定性结论）与 `shortAction`（带价位/仓位的定量动作）**不重复**：一定性一量化
5. UI 展示：波段建议行所有用户可见；短期建议行仅持仓用户可见，未配置时**不渲染**（不留空占位），控制卡片密度

**约束**：
- `bandAction` 锚定校验失败（未引用关键位）→ 视为解析失败，走 Raw 分支，避免 AI 输出不可追溯的观点
- 两字段均为 AI 生成内容，**不构成投资建议**；Footer 合规声明覆盖二者
- Token 成本：每只证券多生成 1 段波段建议，20 只全池串行调用成本略增，可接受（`temperature:0` + 指纹去重已控成本）

**v2.6 推翻（ADR-008 局部作废）**：
- **原因**：v2.5 设计 `bandAction` 全员可见，但无持仓用户处于**观望状态**，需要的是"该不该建仓/在什么价位建仓"（建仓建议），而非"现有持仓怎么操作"（波段建议）。给观望用户看波段建议是**答非所问**。
- **变更**：无持仓时不再输出 `bandAction`，改为输出 `buildPositionAdvice`（建仓建议），同样锚定关键位；有持仓时 `bandAction` 仍保留。
- **ADR-008 第 3、5 条作废**；第 1、2、4 条保留。

### 架构决策：Win11 桌面悬浮与全池刷新（ADR-006）

**背景**：用户希望 AguMaster 以半透明小窗悬浮于 Win11 桌面，展示钉选 1–4 只证券的**完整诊断信息**，同时保留现有最多 10 只自选与对齐调度能力。

**冲突**：
- **展示子集（1–4）** vs **调度全池（10）**：若仅刷钉选可省 Token，但与完整看板数据不一致；若全池刷新则悬浮窗 Token 成本与 H5 相同
- **浏览器 H5** 无法实现真正桌面透明置顶窗，需 Electron 壳
- 功能 3 原「后台 Tab 错过对齐点立即补跑」与部分用户偏好（仅重算、不补跑）不一致

**决策（v1.8 / v2.0）**：
1. **Win11 + Electron** 为桌面悬浮唯一 v1.8 实现路径；H5 浏览器版继续维护
2. 悬浮窗 **仅展示** `widgetPinnedCodes`（1–4 只）；**自动调度与「立即刷新」仍全池 10 只**
3. 单卡 ↻ 与全池自动刷新**解耦**：scheduler /「立即刷新」仍全池；卡片 ↻ 仅刷当前证券（**v2.0**）
4. 托盘/失焦/最小化 **不** 缩小全池自动刷新范围；钉选仅影响 UI 密度
5. **浏览器 H5** 后台 Tab：回到前台 **仅重算** 下一对齐点，**不** 补跑错过轮次；**Electron** 主进程计时不受 Tab 节流影响
6. 悬浮窗卡片 **不再使用紧凑变体**（v1.9）；与完整看板同组件、同配色、同字段，仅只读

**理由**：全池刷新保证悬浮窗与完整看板数据一致、切换视图无延迟；Token 成本可预期且与 v1.7 相同；Electron 在 Win11 透明窗成熟度可接受。

### 架构决策：高频价格监控（ADR-007 / v2.4 重新设计）

**背景**：用户要求在 AI 给出关键位后，能实时监控价格突破并弹窗通知。v2.2 采用的「智能降频」方案根据主调度档位自动启停 30s 轮询，但用户反馈「不够直观、不易理解」。

**决策（v2.4 修订）**：
1. **三态刷新开关**：每只证券可设 `refreshMode`（`off` / `normal` / `overclock`），由用户手动切换
2. **超频上限**：同时最多 **5 只**处于超频状态，防止请求量失控
3. **超频需关键位**：切换至超频要求已配置至少 1 个关键位（无关键位时超频无意义）
4. **独立轮询**：只要有任意证券处于超频且交易时段内，即开启 30s 独立轮询（仅拉行情 + 预警，不触发 AI）
5. **与主调度解耦**：超频轮询不受 `refreshFrequency` 档位影响，用户完全自主控制

**理由**：用户自主控制比自动逻辑更透明；上限 5 只（× 4.5h × 120 轮/小时 = 每日额外约 2700 次）与 v2.2 的峰值请求量相当。

### 术语表
- **快照指纹**：由关键行情字段 + session 拼接的稳定字符串，用于判断是否需要重新诊断
- **自选证券**：自选池内的 A 股股票或场内 ETF/LOF
- **场内基金**：在交易所上市的 ETF/LOF，代码如 `sh510500`、`sz159915`
- **场外基金**：银行/代销渠道申购的开放式基金，v1.4 不支持
- **instrumentType**：`stock` | `fund_etf`，标识快照与 Prompt 资产类型
- **实时快照**：某一时刻的 Latest Quote，非 K 线序列
- **对齐时刻**：以 09:30 为起点的 30 分钟网格时间点；`refreshFrequency` 决定启用子集
- **refreshFrequency**：自动刷新档位，枚举 `5` | `15` | `30` | `60`，默认 `30`
- **pnlPct**：浮动盈亏率，`(现价 - 成本) / 成本 × 100%`
- **shortAction**：有持仓时 LLM 返回的短期持仓动作（≤30 字，原 `action`，v2.5 重命名）
- **bandAction**：有持仓时 LLM 返回的波段操作建议（≤30 字，v2.5 新增，v2.6 改为仅持仓时必填）
- **buildPositionAdvice**：无持仓时 LLM 返回的建仓建议（≤30 字，v2.6 新增，须锚定 supportLevel/resistanceLevel）
- **widgetPinnedCodes**：桌面悬浮窗钉选展示的证券代码列表（1–5 只，须为自选池子集）
- **widgetMode**：`StockCard` 悬浮窗只读模式（隐藏删除/持仓编辑，内容与配色同完整看板）
- **全池刷新**：一轮刷新处理自选池全部证券（最多 20 只），与钉选展示数量无关
- **JSONP**：通过动态 `<script>` 标签跨域获取数据的浏览器技术
- **QuoteProvider**：行情源抽象接口，v1 实现 Tencent / Sina，预留 EastmoneyProxy
- **交易日**：A 股实际开市日；含调休补班日，排除法定放假与普通周末
- **isOffDay**：holiday-cn 字段；`true` = 放假，`false` = 调休补班
- **关键位（keyLevel）**：AI 计算或用户手动设置的价格警戒线；每只证券最多 4 个（AI 支撑、AI 压力、用户自定义 × 2）
- **keyLevelsLocked**：用户锁定关键位后 AI 不再自动更新，防止诊断波动引起关键位频繁跳变
- **偏差保护**：AI 新输出的关键位与旧值偏差超过 ±3% 时暂停更新，弹出确认通知让用户决策
- **穿透判定**：价格从上向下或从下向上穿过关键位的判定逻辑，避免「触碰但不穿透」产生误报
- **智能降频**：主调度档位 ≤ 15 分钟时复用行情结果（蹭车），仅在 30/60 分钟档位开启 30s 独立轮询
- **alertSettings**：桌面通知总开关与各类预警的独立启用/禁用配置
- **固定高风险词库**：`止损` / `逢高减仓` / `减仓` / `清仓` —— 出现时触发 signal 弹窗
- **BrowserWindow 弹窗**：Electron 主进程创建的独立通知窗口，区别于悬浮窗和完整看板主窗
